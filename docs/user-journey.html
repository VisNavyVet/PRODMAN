Agentic Benefits
ProdMan is built agent-first. The primary output of /pm-ff is not a human archive — it's a structured context package for an AI agent about to build. Here's why that matters.
Principle 01
Bounded Scope
Agent knows exactly where to start, where to stop, and what is explicitly out of scope. No interpretation required.
Agent receives a PRD, expands scope, adds features it thinks are related, and ships something 40% larger than planned.
In-scope list, out-of-scope list, and a "must NOT do" section. Agent respects the boundaries because they're explicit.
Principle 02
Testable Acceptance Criteria
Every requirement has observable, verifiable criteria. The agent can check its own output against the spec before it delivers.
AC says "users should be able to find their orders easily." Agent can't verify what "easily" means. Ships and hopes.
AC says "Order history loads in under 1s for the last 90 days. Filter by status returns correct results." Agent can test it.
Principle 03
Embedded Product Context
Agent understands the why — what the product is, who uses it, and what decisions have already been made — without being told.
Agent implements a feature without knowing the user segment, prior API constraints, or that a similar approach was ruled out last quarter.
product.md, users.md, constraints.md, and history.md embedded in the brief. Agent builds for the right user with the right constraints.
Principle 04
Explicit "Must NOT Do"
Overengineering and scope creep happen when agents fill gaps with assumptions. An explicit stop list prevents this.
Agent adds authentication changes, database migrations, and admin UI "because they seem related." Sprint blows up.
must NOT: modify the auth system, touch user schema, or add new API endpoints beyond those listed. Agent stays in lane.
Principle 05
Clear Escalation Triggers
Agent knows exactly when to stop and ask vs. when to proceed. Prevents silent failure and unauthorized decisions.
Agent hits an ambiguity, makes a decision, and ships. PM finds out three days later that the wrong path was taken.
"Stop and ask if: the data model requires a new table, an edge case isn't covered by AC, or the approach differs from approach.md."
Principle 06
Reproducible Context Package
Any agent — new session, different model, mid-feature handoff — can pick up the same brief and continue without losing context.
Context lives in a long chat thread. New session or new agent means starting over or pasting walls of text to catch up.
features/[name]/agent-brief.md is a self-contained context package. Any agent, any session, any model — consistent output.
Agent brief anatomy — what each field does for the agent
task_summary
1–3 sentence orientation. Agent knows the job before reading anything else.
product_context
What the product does, who uses it. Agent builds for the right user.
technical_context
Stack, patterns, constraints. Agent doesn't contradict your architecture.
in_scope / out_of_scope
Hard boundaries. Agent knows exactly what to build and what to leave alone.
must_not_do
Explicit stop list. Prevents overengineering and unauthorized scope expansion.
acceptance_criteria
Observable, verifiable conditions. Agent self-checks before delivering.
edge_cases
Pre-enumerated. Agent handles them consistently, not by guessing.
constraints
Style, performance, accessibility, security. Non-negotiable requirements.
definition_of_done
Clear completion signal. Agent knows when to stop, not just when to ship.
escalation_triggers
Explicit "stop and ask" conditions. No silent failures or unauthorized decisions.
