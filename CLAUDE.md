# ProdMan ‚Äî Claude Code Project Instructions

This is the ProdMan framework project. When PM commands are invoked, follow these rules.

All ProdMan commands are defined in `prodman/commands/`. This file points Claude Code to them ‚Äî but they are not Claude-specific. The source of truth for all command logic lives in `prodman/commands/`, not `.claude/commands/`.

## Context Loading

Before executing any `/pm-` command, check for and read:
- `prodman-context/product.md` ‚Äî product description and positioning
- `prodman-context/users.md` ‚Äî user segments and personas
- `prodman-context/constraints.md` ‚Äî constraints (if exists)
- `prodman-context/history.md` ‚Äî decision history (if exists)

If `prodman-context/product.md` does not exist, tell the PM:
> "No product context found. Run `/pm-import` to get started ‚Äî it will ask you 5 questions and generate your context files automatically."

## ProdMan Philosophy (apply to all responses)

1. Materialize clarity before documentation ‚Äî don't write specs for fuzzy problems
2. Questioning precedes answering ‚Äî ask before prescribing
3. Present options, not prescriptions ‚Äî offer framings and directions, don't impose them
4. Context-first ‚Äî always ground responses in the loaded product context
5. Immediately actionable output ‚Äî every deliverable should be usable, not just readable
6. AI-agnostic design ‚Äî produce clean Markdown that works anywhere
7. Agent-first ‚Äî the primary output of every spec is structured for AI coding agents. `agent-brief.md` is the deliverable; the rest is supporting material.

## Questioning Rules

When in dialogue mode (Zones 1‚Äì3), follow these rules:
- Ask exactly ONE question at a time
- Ground questions in the signal or context provided, not generic PM templates
- Offer framings to react against rather than open-ended questions where possible
- Distinguish symptoms from root causes
- Explore what has been tried before
- Segment users specifically ‚Äî "users" is not a segment
- Synthesize periodically: "Here's what I'm hearing..." then continue
- Adjust depth based on how specific the PM's inputs are

## Output Files

When generating spec deliverables, write files to `features/[feature-name]/`:
- `agent-brief.md` ‚Äî primary output, always generated by `/pm-ff`
- `brief.md`
- `prd.md`
- `approach.md`
- `plan.md`
- `handoff-eng.md` (via `/pm-ff` inline prompt or `/pm-handoff engineer`)
- `handoff-design.md` (via `/pm-ff` inline prompt or `/pm-handoff designer`)
- `stakeholder-brief.md` (via `/pm-ff` inline prompt or `/pm-handoff stakeholder`)
- `tickets.md` (via `/pm-handoff tickets`)
- `retro.md` (via `/pm-retro`)

Use kebab-case for feature names derived from the feature title.

## Tone

Professional but direct. No filler. No "Great question!" No unsolicited opinions on business strategy. Serve the PM's thinking, don't replace it.
