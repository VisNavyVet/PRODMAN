**Maintained by:** ProdMan contributors

## Overview

`prodman-spec` is an open standard for AI-consumable product specifications. It defines the structure, required fields, and quality criteria that a spec bundle must meet to be considered **agent-ready** — meaning an AI coding agent can act on it without requiring clarification from a human.

Any tool can produce or consume a prodman-spec. It is AI-agnostic, tool-agnostic, and language-agnostic.

---

## Motivation

AI coding agents fail not because they're incapable, but because the specs they receive are:
- Missing scope boundaries (agents fill in gaps with assumptions)
- Untestable (no way to know when the task is done)
- Underspecified on edge cases (agents pick a path, often the wrong one)
- Missing escalation triggers (agents make unauthorized decisions)

`prodman-spec` is a contract between the PM and the agent. When a spec conforms to this standard, agents can build correctly on the first attempt.

---

## Spec Bundle

A spec bundle is a directory of Markdown files for a single feature. The standard defines two tiers:

### Tier 1 — Minimal (required for `agent-ready`)

| File | Purpose |
|------|---------|
| `agent-brief.md` | The primary agent input. Contains all required fields (see below). |

### Tier 2 — Full bundle (generated by `/pm-ff`)

| File | Purpose |
|------|---------|
| `agent-brief.md` | Primary agent input |
| `brief.md` | Problem framing and success metrics for PMs and stakeholders |
| `prd.md` | Full requirements, user stories, edge cases, open questions |
| `approach.md` | Technical direction, options considered, risk register |
| `plan.md` | Delivery phases, milestones, launch checklist |

The full bundle provides context for human reviewers. Only `agent-brief.md` is required by this spec.

## Required Fields — agent-brief.md

A conforming `agent-brief.md` must contain all of the following sections. Absent or empty sections cause a spec to be marked `incomplete`.

| Section | Heading | Description |
|---------|---------|-------------|
| Task Summary | `## Task Summary` | 1–2 sentences. What the agent should build or change. Outcome-focused, not implementation-focused. |
| Product Context | `## Context / ### Product context` | Product description and affected user segment. |
| Technical Context | `## Context / ### Technical context` | Repo, relevant files, tech stack, related systems. |
| In Scope | `## Task Scope / ### In scope` | Specific deliverables. Minimum 1 item. |
| Out of Scope | `## Task Scope / ### Out of scope` | Explicit exclusions. Minimum 1 item. |
| Requirements | `## Requirements / ### Must implement` | Specific, testable requirements. Minimum 1 item. |
| Anti-Requirements | `## Requirements / ### Must NOT do` | Things the agent must not do. Minimum 1 item. |
| Acceptance Criteria | `## Acceptance Criteria` | Testable conditions. Minimum 3 items. Each must be observable. |
| Edge Cases | `## Edge Cases to Handle` | Scenarios with explicit expected behavior. |
| Constraints | `## Constraints` | Performance, accessibility, security constraints. |
| Definition of Done | `## Definition of Done` | Checklist. Minimum 3 items. |
| Escalation Triggers | `## Questions the Agent Should Flag` | Conditions under which the agent must stop and ask rather than decide. |

---

## Quality Levels

### `agent-ready`
All required sections are present and non-empty. Acceptance criteria are testable (observable behavior, not vague outcomes). Scope is bounded. At least one escalation trigger is defined.

### `review-needed`
All required sections are present, but one or more quality warnings exist:
- Acceptance criteria contain vague language ("works correctly", "looks good", "is fast")
- Out of scope has fewer than 2 items
- No escalation triggers defined

An agent can proceed with a `review-needed` spec, but the PM should acknowledge the warnings.

### `incomplete`
One or more required sections are missing or empty. An agent must not begin work on an `incomplete` spec.

## Compiled Spec (machine-readable)

A prodman-spec can be compiled into `compiled-spec.json` — a structured JSON document that directly maps to the sections above. The JSON format is defined in `schemas/compiled-spec-schema.json`.

The compiled spec is the primary format for programmatic consumption: CI validation, agent launchers, observability dashboards, and integration with project management tools.

**Compilation:** `prodman compile [feature-name]`
**Validation:** `prodman validate [feature-name]`
**Schema:** [`schemas/compiled-spec-schema.json`](../schemas/compiled-spec-schema.json)

## Linter Rules

The following rules are checked by `prodman validate`. Rules marked `error` block the `agent-ready` state. Rules marked `warning` produce `review-needed`.

| ID | Severity | Section | Rule |
|----|----------|---------|------|
| LNT-001 | error | Task Summary | Section is present and non-empty |
| LNT-002 | error | Task Summary | Length is 10–500 characters |
| LNT-003 | error | Technical Context | `relevant_files` contains at least 1 entry |
| LNT-004 | error | In Scope | At least 1 item present |
| LNT-005 | error | Out of Scope | At least 1 item present |
| LNT-006 | error | Requirements | At least 1 item present |
| LNT-007 | error | Anti-Requirements | At least 1 item present |
| LNT-008 | error | Acceptance Criteria | At least 3 items present |
| LNT-009 | warning | Acceptance Criteria | No item contains vague phrases: "works correctly", "looks good", "is fast", "handles errors", "as expected" |
| LNT-010 | error | Definition of Done | At least 3 items present |
| LNT-011 | warning | Escalation Triggers | At least 1 trigger is defined |

## Versioning

This document is versioned. Breaking changes (new required fields, removed fields) increment the major version. The `$schema` field in `compiled-spec.json` identifies the spec version used during compilation.

- **v1.0.0** (current) — initial published standard. 12 required JSON fields. 11 lint rules.

---

## Conformance

A tool claiming `prodman-spec v1` conformance must:
1. Accept a spec bundle directory as input
2. Validate all 11 lint rules
3. Produce a `readiness` state of `agent-ready`, `review-needed`, or `incomplete`
4. Produce a `compiled-spec.json` conforming to `schemas/compiled-spec-schema.json` when asked
